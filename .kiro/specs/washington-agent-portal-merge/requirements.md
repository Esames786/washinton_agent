# Requirements Document

## Introduction

This project produces **two new independent portals** that are completely isolated from the existing live portals. The old portals (`daydispatchagent`, `daydispatchhr`, `washinton_latest`) are never modified.

**New portals:**
- **`washinton_agent`** → `washintonagent.shawntransport.com` — Washington CRM with DayDispatch public frontend merged in
- **`washinton_hr`** → `hrwashinton.shawntransport.com` — Copy of the HR portal, linked to Washington Agent

Both new portals share a **single database: `shiap16_main2`** (the existing Washington database on the same cPanel server). HR portal tables are added to this database with an `hr_` prefix to avoid name collisions with Washington's existing tables.

## Glossary

- **Washington_Portal**: The new merged portal codebase (`washinton_agent`), derived from `washinton_latest`, running at `washintonagent.shawntransport.com`.
- **Washington_HR_Portal**: The new HR portal codebase (`washinton_hr`), derived from `daydispatchhr`, running at `hrwashinton.shawntransport.com`.
- **Old_Agent_Portal**: The existing `daydispatchagent` portal at `agent.daydispatch.com` — untouched.
- **Old_HR_Portal**: The existing `daydispatchhr` portal at `hragent.daydispatch.com` — untouched.
- **Shared_Database**: The `shiap16_main2` MySQL database used by both new portals.
- **HR_Prefix**: The `hr_` prefix applied to all Washington_HR_Portal table names (e.g. `hr_employees`, `hr_payrolls`) to prevent collisions with Washington_Portal tables in the Shared_Database.
- **Washington_User**: A user authenticated in the Washington_Portal, stored in the `user` table of the Shared_Database.
- **HR_Employee**: An employee record in the Washington_HR_Portal, stored in the `hr_employees` table of the Shared_Database.
- **Agent_ID**: The integer identifier stored in `user.agent_id` (Washington_Portal) and `hr_employees.agent_id` (Washington_HR_Portal) that links a Washington_User to an HR_Employee.
- **Bridge_Service**: The HTTP client in Washington_Portal that communicates with Washington_HR_Portal bridge endpoints using a shared secret key (`X-Bridge-Key` header).
- **Bridge_Key**: The shared secret string configured in `config/bridge.php` used to authenticate inter-portal HTTP requests between the two new portals.
- **SSO_Redirect**: A time-limited, cryptographically signed URL generated by Washington_HR_Portal that auto-logs in an HR_Employee when visited.
- **Commission_Screen**: The new Washington_Portal screen that displays per-user order completion, payment, profit, and commission earned data.
- **Public_Frontend**: The public website pages (About, FAQ, Terms, Carriers, Brokers, Quote Request, etc.) copied from the Old_Agent_Portal into the Washington_Portal, converted from Livewire to Blade.
- **Quote_Request_Form**: The public-facing lead capture form at `/Quote-Request` on the Public_Frontend.
- **View_Query_Screen**: The existing Washington_Portal screen at `/view_query` that displays and manages incoming leads.
- **Profit_Table**: The `profit` table in the Shared_Database storing profit data per order.
- **Order_Table**: The `order` table in the Shared_Database storing transport orders.
- **Commission_Range**: The `commission_ranges` table in the Shared_Database defining commission tiers by profit range.
- **Panel_Permission**: Washington_Portal's existing role-based access control system using panel types.
- **Signup_Type**: The user-selected account type on the public signup form — either "Agent" (maps to Order Taker role) or "Carrier" (maps to Dispatcher role).
- **Reference_User_130**: The existing Washington_User with id=130, used as the permission template for all Agent signups.
- **Reference_User_53**: The existing Washington_User with id=53, used as the permission template for all Carrier/Dispatcher signups.
- **Inactive_Status**: A Washington_User account state (`status = 0`) — account cannot access portal screens until an admin activates it.

## Requirements

### Requirement 1: Washington Portal Codebase Setup

**User Story:** As a system administrator, I want the `washinton_agent` portal initialised from Washington and the `washinton_hr` portal initialised from the HR portal, both pointing to the same `shiap16_main2` database, so that the two new portals are fully independent from the old live portals.

#### Acceptance Criteria

1. THE `washinton_agent` folder SHALL be initialised from a complete copy of the `washinton_latest` codebase.
2. THE `washinton_hr` folder SHALL be initialised from a complete copy of the `daydispatchhr` codebase.
3. THE Washington_Portal SHALL connect to `shiap16_main2` as its primary database.
4. THE Washington_HR_Portal SHALL connect to `shiap16_main2` as its primary database.
5. THE Washington_HR_Portal SHALL use `hr_` prefixed table names for all its tables (e.g. `hr_employees`, `hr_payrolls`, `hr_departments`) so that no table name collides with Washington_Portal's existing tables.
6. ALL Eloquent models in `washinton_hr` SHALL have their `$table` property updated to include the `hr_` prefix.
7. THE Washington_HR_Portal Spatie permission config SHALL reference `hr_` prefixed permission tables.
8. WHEN the Washington_Portal is deployed, it SHALL serve all existing Washington CRM routes and screens without regression.
9. WHEN the Washington_HR_Portal is deployed, it SHALL serve all existing HR portal routes and screens without regression.
10. THE `daydispatchagent`, `daydispatchhr`, and `washinton_latest` folders and their databases SHALL NOT be modified in any way.

---

### Requirement 2: Public Frontend Website Integration

**User Story:** As a potential customer, I want to access the DayDispatch public website pages through the Washington_Portal domain, so that there is a single public-facing entry point for the business.

#### Acceptance Criteria

1. THE Washington_Portal SHALL serve all public frontend pages copied from the Old_Agent_Portal, including: Home, About, FAQ, Terms of Service, Carriers, Brokers, and Blog pages.
2. THE Washington_Portal SHALL serve the Quote_Request_Form at the `/Quote-Request` route.
3. WHEN a visitor submits the Quote_Request_Form, THE Washington_Portal SHALL store the submission and make it visible in the View_Query_Screen at `/view_query`.
4. WHEN a visitor submits the Quote_Request_Form, THE Washington_Portal SHALL redirect the visitor to a confirmation page within the Washington_Portal domain.
5. THE Washington_Portal SHALL NOT forward Quote_Request_Form submissions to the Old_Agent_Portal's lead handling endpoints.
6. THE Washington_Portal SHALL render all public frontend pages without requiring authentication.
7. IF a public frontend page asset (image, CSS, JS) is missing after migration, THEN THE Washington_Portal SHALL log a 404 error and serve a graceful fallback rather than a blank page.

### Requirement 3: Public Self-Service Signup

**User Story:** As a new agent or carrier, I want to register for a Washington_Portal account from the public signup page, so that I can request access without contacting an administrator.

#### Acceptance Criteria

1. THE Washington_Portal SHALL provide a public signup page at `/register` that does not require authentication to access.
2. THE signup form SHALL collect exactly: Full Name, Email, Password, Phone Number, and Full Address.
3. THE signup form SHALL present a Signup_Type selector with exactly two options: **"Agent"** and **"Carrier"**. No other type options SHALL appear.
4. WHEN a visitor selects "Agent", THE Washington_Portal SHALL map this to the Order Taker role.
5. WHEN a visitor selects "Carrier", THE Washington_Portal SHALL map this to the Dispatcher role.
6. WHEN a visitor submits the signup form with valid data, THE Washington_Portal SHALL create a new `user` record with: `name`, `email`, `password` (bcrypt), `phone`, `address`, `role` (from Signup_Type), `status = 0`.
7. WHEN a new user registers as "Agent", THE Washington_Portal SHALL copy all permission fields from Reference_User_130 to the new user record.
8. WHEN a new user registers as "Carrier", THE Washington_Portal SHALL copy all permission fields from Reference_User_53 to the new user record.
9. WHEN a new user registers as "Agent", THE Washington_Portal SHALL create a `user_settings` record with `penal_type = 1`.
10. WHEN a new user registers as "Carrier", THE Washington_Portal SHALL create a `user_settings` record with the `penal_type` matching Reference_User_53's setting.
11. AFTER a successful signup, THE Washington_Portal SHALL redirect to the login page with a pending-activation message.
12. WHEN a visitor submits with a duplicate email, THE Washington_Portal SHALL display a validation error and SHALL NOT create a duplicate record.
13. WHEN a visitor submits with a missing required field, THE Washington_Portal SHALL display a field-level validation error and SHALL NOT create a user record.
14. THE Washington_Portal SHALL NOT send any automated activation email.
15. THE existing bridge signup and login endpoints SHALL continue to function and SHALL write new users to the `user` table using the same field mapping and permission-copy logic.

### Requirement 4: Agent ID Field on Washington User

**User Story:** As an administrator, I want to assign an HR portal Agent ID to each Washington user, so that Washington users can be linked to their corresponding HR employee records.

#### Acceptance Criteria

1. THE Washington_Portal SHALL add a nullable integer `agent_id` column to the `user` table via a database migration.
2. THE Washington_Portal SHALL display an "Agent ID" input field on the Edit Employee screen.
3. WHEN an administrator saves an Agent ID, THE Washington_Portal SHALL persist the value to `user.agent_id`.
4. WHEN an administrator saves an Agent ID, THE Washington_Portal SHALL validate that the value is a positive integer or null.
5. IF an administrator enters a non-integer or negative value, THEN THE Washington_Portal SHALL display a validation error and SHALL NOT save the invalid value.
6. THE Washington_Portal SHALL allow the Agent ID field to be cleared (set to null) to unlink a user.
7. THE Washington_Portal SHALL enforce that the same `agent_id` is not assigned to more than one Washington_User (unique constraint, nullable).

### Requirement 5: HR Portal SSO — Washington Side

**User Story:** As a Washington admin, I want to click "Open HR Portal" on the Edit Employee screen and be automatically logged into the Washington_HR_Portal as the linked employee, so that I do not need separate HR credentials.

#### Acceptance Criteria

1. THE Washington_Portal SHALL include `config/bridge.php` with the Washington_HR_Portal base URL (`hrwashinton.shawntransport.com`), Bridge_Key, and endpoint paths.
2. THE Washington_Portal SHALL include a Bridge_Service class that sends authenticated HTTP POST requests to Washington_HR_Portal using the `X-Bridge-Key` header.
3. WHEN an administrator views the Edit Employee screen for a user with a non-null `agent_id`, THE Washington_Portal SHALL display an "Open HR Portal" button.
4. WHEN the button is clicked, THE Bridge_Service SHALL call `POST /bridge/agent/login` on Washington_HR_Portal with the user's `agent_id`.
5. WHEN Washington_HR_Portal returns a valid `redirect_url`, THE Washington_Portal SHALL redirect the browser to that SSO_Redirect URL.
6. IF Washington_HR_Portal returns an error, THEN THE Washington_Portal SHALL display a descriptive error message and SHALL NOT redirect.
7. IF the Bridge_Key is blank, THEN THE Bridge_Service SHALL throw a `RuntimeException` before making any HTTP request.
8. THE Bridge_Service SHALL set a 20-second HTTP timeout on all requests.
9. THE Washington_Portal SHALL NOT expose the Bridge_Key in any HTTP response or frontend asset.

---

### Requirement 6: HR Portal SSO — Washington_HR_Portal Side

**User Story:** As a system integrator, I want the Washington_HR_Portal to accept SSO login requests from Washington_Portal using `agent_id`, so that the bridge mechanism works correctly.

#### Acceptance Criteria

1. THE Washington_HR_Portal SHALL accept `POST /bridge/agent/login` requests authenticated with the `X-Bridge-Key` header.
2. WHEN a valid request is received for an `agent_id` matching an active `hr_employees` record, THE Washington_HR_Portal SHALL return a `redirect_url` signed URL valid for no more than 2 minutes.
3. WHEN a valid request is received for an `agent_id` with no linked employee, THE Washington_HR_Portal SHALL return a 404 JSON response.
4. WHEN a valid request is received for an `agent_id` whose employee is inactive, THE Washington_HR_Portal SHALL return a 403 JSON response.
5. WHEN a browser visits the SSO_Redirect URL within 2 minutes, THE Washington_HR_Portal SHALL authenticate the employee session and redirect to the employee dashboard.
6. IF the SSO_Redirect URL is visited after 2 minutes, THEN THE Washington_HR_Portal SHALL return 403 and SHALL NOT authenticate the session.

### Requirement 7: Commission and Profit Screen

**User Story:** As a manager, I want to view a per-user commission summary screen showing orders completed, payment collected, profit earned, and commission due for any date range.

#### Acceptance Criteria

1. THE Washington_Portal SHALL provide a Commission_Screen accessible to users with the appropriate Panel_Permission.
2. THE Commission_Screen SHALL display one row per Washington_User with: user name, completed order count (pstatus=13), total payment, total profit, and calculated commission for the selected period.
3. WHEN a manager selects a date range filter, THE Washington_Portal SHALL recalculate and display data for only orders completed within that range.
4. WHEN a manager selects a month/year filter, THE Washington_Portal SHALL recalculate and display data for only that calendar month.
5. THE Washington_Portal SHALL calculate commission by joining Order_Table, Profit_Table, and Commission_Range using the profit value to determine the applicable tier.
6. THE Commission_Screen SHALL display zero values for users with no completed orders in the period rather than omitting them.
7. THE Commission_Screen SHALL be filterable by individual user to show a per-order breakdown.
8. WHEN a manager views the per-order breakdown, THE Washington_Portal SHALL display: order ID, completion date, payment, profit, and commission per order.
9. THE Commission_Screen SHALL display all monetary values in USD with two decimal places.

---

### Requirement 8: Washington Bridge Endpoints for HR Portal

**User Story:** As the Washington_HR_Portal system, I want to call Washington_Portal bridge endpoints to retrieve commission data and user status for a given `agent_id`, so that HR payroll can incorporate Washington order commissions.

#### Acceptance Criteria

1. THE Washington_Portal SHALL expose `POST /bridge/washington/agent/commission` authenticated by `X-Bridge-Key`.
2. WHEN a valid request is received with `agent_id` and optional `month` (YYYY-MM), THE Washington_Portal SHALL return: `agent_id`, `user_id`, `user_name`, `period`, `completed_orders_count`, `total_payment`, `total_profit`, `total_commission`.
3. WHEN a valid request is received for an unknown `agent_id`, THE Washington_Portal SHALL return a 404 JSON response.
4. THE Washington_Portal SHALL expose `POST /bridge/washington/agent/status` authenticated by `X-Bridge-Key`.
5. WHEN a valid request is received with `agent_id`, THE Washington_Portal SHALL return: `linked` (boolean), `active` (boolean), `user_id`, `user_name`, `message`.
6. WHEN a valid request is received for an unknown `agent_id`, THE Washington_Portal SHALL return `linked: false`, `active: false`.
7. IF the `X-Bridge-Key` header is missing or invalid, THEN THE Washington_Portal SHALL return 401 and SHALL NOT return any data.
8. IF `agent_id` is not a positive integer, THEN THE Washington_Portal SHALL return 422 with a validation error.

---

### Requirement 9: Commission Data Integrity

**User Story:** As a payroll administrator, I want commission calculations to be consistent and reproducible so that the same input always produces the same output.

#### Acceptance Criteria

1. THE Washington_Portal SHALL calculate commission by looking up the profit value in Commission_Range and applying the matching tier.
2. WHEN the same orders and profit values are provided, THE Commission_Screen SHALL display the same total on every page load.
3. WHEN a commission calculation is requested via bridge endpoint for the same `agent_id` and `month`, THE Washington_Portal SHALL return the same total on every call.
4. THE Washington_Portal SHALL NOT modify any order, profit, or payment record as a side effect of commission calculation.
5. IF no Commission_Range row matches a profit value, THEN THE Washington_Portal SHALL treat that order's commission as zero and SHALL log a warning with the profit value and order ID.

---

### Requirement 10: Security and Access Control

**User Story:** As a security administrator, I want all bridge communications and sensitive screens to be properly authenticated and access-controlled.

#### Acceptance Criteria

1. THE Washington_Portal SHALL validate the `X-Bridge-Key` header on all `/bridge/washington/agent/*` endpoints before executing any business logic.
2. THE Commission_Screen SHALL be accessible only to authenticated Washington_Users with the commission Panel_Permission.
3. THE "Open HR Portal" button SHALL be visible only to Washington_Users with the employee management Panel_Permission.
4. THE Washington_Portal SHALL NOT store any Bridge_Key in the database, session, or any client-side variable.
5. WHEN the Bridge_Service receives an HTTP error from Washington_HR_Portal, THE Washington_Portal SHALL log the error (excluding secret values) and SHALL NOT expose raw error details to the end user.
6. THE bridge endpoints SHALL accept requests only over HTTPS in production.

---

### Requirement 11: Configuration and Deployment

**User Story:** As a DevOps engineer, I want all inter-portal settings managed through environment variables so the portals can be deployed without code changes.

#### Acceptance Criteria

1. THE Washington_Portal SHALL read Washington_HR_Portal base URL, Bridge_Key, and endpoint paths from `config/bridge.php` sourced from environment variables.
2. THE Washington_Portal `config/bridge.php` SHALL define: `hrportal.base_url` (pointing to `hrwashinton.shawntransport.com`), `hrportal.shared_key`, `hrportal.agent_login_endpoint`, `hrportal.agent_status_endpoint`, and `washington.shared_key`.
3. THE Washington_HR_Portal `config/bridge.php` SHALL define its own `shared_key` (the key used to authenticate inbound requests from Washington_Portal) — separate from the Old_HR_Portal's key.
4. BOTH new portals' `.env.example` files SHALL include placeholder entries for all bridge-related environment variables.
5. WHEN a required bridge environment variable is missing or blank at runtime, THE Bridge_Service SHALL throw a `RuntimeException` identifying the missing variable before making any HTTP request.

### Requirement 6: HR Portal SSO Bridge — HR Side Compatibility

3. WHEN the HR_Portal receives a valid `agent_login` request for an `agent_id` that has no linked HR_Employee, THE HR_Portal SHALL return a 404 JSON response with a descriptive `message` field.
4. WHEN the HR_Portal receives a valid `agent_login` request for an `agent_id` whose linked HR_Employee has `employee_status_id` not equal to 1 (inactive), THE HR_Portal SHALL return a 403 JSON response with a descriptive `message` field.
5. WHEN a browser visits the SSO_Redirect URL within the 2-minute validity window, THE HR_Portal SHALL authenticate the HR_Employee session and redirect to the employee dashboard.
6. IF a browser visits the SSO_Redirect URL after the 2-minute validity window has expired, THEN THE HR_Portal SHALL return a 403 response and SHALL NOT authenticate the session.

### Requirement 7: Commission and Profit Screen

**User Story:** As a manager, I want to view a per-user commission summary screen in the Washington_Portal showing orders completed, total payment collected, profit earned, and commission due for any date range, so that I can review agent performance and feed data to HR payroll.

#### Acceptance Criteria

1. THE Washington_Portal SHALL provide a Commission_Screen accessible to users with the appropriate Panel_Permission.
2. THE Commission_Screen SHALL display a summary table with one row per Washington_User containing: user name, count of completed orders (pstatus = 13), total payment amount, total profit amount, and calculated commission amount for the selected period.
3. WHEN a manager selects a date range filter on the Commission_Screen, THE Washington_Portal SHALL recalculate and display commission data for only the orders completed within that date range.
4. WHEN a manager selects a month/year filter on the Commission_Screen, THE Washington_Portal SHALL recalculate and display commission data for only the orders completed within that calendar month.
5. THE Washington_Portal SHALL calculate commission amounts by joining the Order_Table, Profit_Table, and Commission_Range table using the profit value to determine the applicable commission tier.
6. THE Commission_Screen SHALL display a zero commission value for Washington_Users who have no completed orders in the selected period rather than omitting those users from the table.
7. THE Commission_Screen SHALL be filterable by individual Washington_User to show only that user's order-level commission breakdown.
8. WHEN a manager views the per-user order breakdown, THE Washington_Portal SHALL display: order ID, order completion date, payment amount, profit amount, and commission amount for each order.
9. THE Commission_Screen SHALL display all monetary values in USD with two decimal places.

### Requirement 8: Washington Bridge Endpoints for HR Portal

**User Story:** As the HR portal system, I want to call Washington_Portal bridge endpoints to retrieve commission data and user status for a given `agent_id`, so that HR payroll can incorporate Washington order commissions without manual data entry.

#### Acceptance Criteria

1. THE Washington_Portal SHALL expose a `POST /bridge/washington/agent/commission` endpoint authenticated by the `X-Bridge-Key` header.
2. WHEN the `POST /bridge/washington/agent/commission` endpoint receives a valid request with an `agent_id` integer and an optional `month` (YYYY-MM format) parameter, THE Washington_Portal SHALL return a JSON response containing: `agent_id`, `user_id`, `user_name`, `period`, `completed_orders_count`, `total_payment`, `total_profit`, and `total_commission`.
3. WHEN the `POST /bridge/washington/agent/commission` endpoint receives a request for an `agent_id` that does not match any Washington_User, THE Washington_Portal SHALL return a 404 JSON response with a descriptive `message` field.
4. THE Washington_Portal SHALL expose a `POST /bridge/washington/agent/status` endpoint authenticated by the `X-Bridge-Key` header.
5. WHEN the `POST /bridge/washington/agent/status` endpoint receives a valid request with an `agent_id` integer, THE Washington_Portal SHALL return a JSON response containing: `linked` (boolean), `active` (boolean), `user_id`, `user_name`, and `message`.
6. WHEN the `POST /bridge/washington/agent/status` endpoint receives a request for an `agent_id` that does not match any Washington_User, THE Washington_Portal SHALL return a JSON response with `linked: false`, `active: false`, and a descriptive `message`.
7. IF a request to either bridge endpoint is missing the `X-Bridge-Key` header or the key does not match the configured Bridge_Key, THEN THE Washington_Portal SHALL return a 401 JSON response and SHALL NOT return any user or commission data.
8. IF a request to either bridge endpoint contains an `agent_id` that is not a positive integer, THEN THE Washington_Portal SHALL return a 422 JSON response with a descriptive validation error.

### Requirement 9: Commission Data Integrity

**User Story:** As a payroll administrator, I want commission calculations in the Washington_Portal to be consistent and reproducible, so that the same input data always produces the same commission output regardless of when the calculation is run.

#### Acceptance Criteria

1. THE Washington_Portal SHALL calculate commission for an order by looking up the profit value in the Commission_Range table and applying the matching tier's rate or fixed amount.
2. WHEN the same set of completed orders and profit values is provided, THE Commission_Screen SHALL display the same total commission value on every page load.
3. WHEN a commission calculation is requested via the bridge endpoint for a given `agent_id` and `month`, THE Washington_Portal SHALL return the same commission total for the same input data on every call (idempotent read).
4. THE Washington_Portal SHALL NOT modify any order, profit, or payment record as a side effect of displaying the Commission_Screen or responding to a bridge commission endpoint request.
5. IF no Commission_Range row matches a given profit value, THEN THE Washington_Portal SHALL treat the commission for that order as zero and SHALL log a warning identifying the unmatched profit value and order ID.

### Requirement 10: Security and Access Control

**User Story:** As a security administrator, I want all inter-portal bridge communications and sensitive screens to be properly authenticated and access-controlled, so that unauthorised parties cannot access commission data, trigger SSO logins, or read user records.

#### Acceptance Criteria

1. THE Washington_Portal SHALL protect all bridge endpoints (`/bridge/washington/agent/*`) with `X-Bridge-Key` header validation before executing any business logic.
2. THE Commission_Screen SHALL be accessible only to Washington_Users whose role includes the commission Panel_Permission; unauthenticated or unauthorised requests SHALL be redirected to the login screen.
3. THE "Open HR Portal" SSO button SHALL be visible only to Washington_Users with the employee management Panel_Permission.
4. THE Washington_Portal SHALL NOT store the Bridge_Key in any database table, session, or client-side variable; it SHALL be read exclusively from the server-side configuration file.
5. WHEN the Bridge_Service receives an HTTP error response from the HR_Portal, THE Washington_Portal SHALL log the error with the HTTP status code and response body (excluding any secret values) and SHALL NOT expose raw error details to the end user.
6. THE Washington_Portal bridge endpoints SHALL accept requests only over HTTPS in production; HTTP requests SHALL be rejected with a 403 response.

### Requirement 11: Configuration and Deployment

**User Story:** As a DevOps engineer, I want all inter-portal integration settings to be managed through environment variables and a dedicated config file, so that the Washington_Portal can be deployed to different environments without code changes.

#### Acceptance Criteria

1. THE Washington_Portal SHALL read the HR_Portal base URL, Bridge_Key, and endpoint paths from `config/bridge.php`, which SHALL source its values from environment variables.
2. THE Washington_Portal `config/bridge.php` SHALL define at minimum: `hrportal.base_url`, `hrportal.shared_key`, `hrportal.agent_login_endpoint`, `hrportal.agent_status_endpoint`, and `washington.shared_key` (the key used to authenticate inbound bridge requests).
3. THE Washington_Portal `.env.example` file SHALL include placeholder entries for all bridge-related environment variables so that new deployments can identify required configuration.
4. WHEN a required bridge environment variable is missing or blank at runtime, THE Bridge_Service SHALL throw a `RuntimeException` with a descriptive message identifying the missing variable before making any HTTP request.
