1. Architecture Overview

OLD — washinton_latest / washinton_agent

Embedded Payment Model

Payment data is stored directly as columns on the order table. No separate payment entity exists. Profit is calculated inline as payment - pay_carrier. There is no approval workflow, no screenshot proof, and no per-payment audit trail. The payment_logs table tracks individual transactions but is not linked to a formal approval process.

Commission Problem: The HR payroll job (GeneratePayrollJob) queries the order_payments table — but this table does NOT exist in shiap16_main2. It only existed in the old shipa15_daydispatch_agent_live database. Commission calculation currently returns 0.0 for all agents silently.

NEW — daydispatchagent

Standalone Payment Entity Model

Payment is a completely separate entity in the order_payments table. Each payment has its own lifecycle: Pending → Return → Confirmed. Agents upload screenshot proof. Admin reviews and confirms or returns for correction. Full audit trail via order_payment_logs. Full status history via order_payment_journeys.

Commission: HR payroll queries order_payments where payment_status = 'Payment Confirmed' and is_paid = 0, sums profit (USD), converts to PKR via hr_currency_rates, applies commission setting (% or flat), then marks rows is_paid = 1 to prevent double-counting.

  • HR commission uses confirmed + unpaid rows
  • After commission run: is_paid=1 prevents double-counting

⚠ Critical Issue — Commission is Broken Right Now

  • The order_payments table was created in shipa15_daydispatch_agent_live (old daydispatch DB)
  • The new shared DB shiap16_main2 does NOT have this table
  • The hr_currency_rates table may also be missing from shiap16_main2
  • Result: All commission calculations return 0.0 silently — no error thrown, just wrong payroll numbers
  • This affects all employees with account_type_id = 2 (commission-only) or 3 (salary + commission)
